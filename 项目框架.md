# 项目框架

## 根目录
- index.html：主页面入口，加载工作面板主体
- wechat-upload.html：微信专用上传识别入口
- css/style.css：全局样式
- supabase_setup.sql：Supabase 初始化脚本
- supabase_setup_safe.sql：更安全的 Supabase 初始化脚本

## js 模块
- js/app.js：主界面交互、视图切换、上传入口、识别预览确认
- js/ocr.js：OCR/AI 识别、会议去重、识别动作计划、确认写入
- js/wechat-upload.js：微信上传页交互、预览确认、保存执行
- js/db.js：本地数据存储与事项增删改查
- js/sync.js：云端同步逻辑
- js/report.js：报告生成与导出
- js/calendar.js：日历相关逻辑
- js/templates.js：模板与渲染辅助
- js/crypto.js：加密相关工具
- js/kimi.js：Kimi 相关能力封装

## 当前重点流程
- 主页面上传：选择文件 → OCR 识别 → 预览确认 → 确认后写入面板
- 微信上传：进入轻量页 → 选择文件 → OCR 识别 → 预览确认 → 确认后写入面板
- 会议识别：标题清洗 → 时间地点归一化 → 已有会议比对 → 新增 / 合并 / 跳过
- 取消兜底：预览前先抓取事项快照，若点击取消则恢复快照，确保不发生误写入

## 当前评估方向
- 高性价比优化优先覆盖上传识别、预览确认、同步触发、微信兼容、导出稳定性
- 当前已进入“安全优化实施阶段”，第一回合实际主题已切换为“Word 导出本地优先、CDN 兜底”，以降低外部 CDN 波动对导出的影响

## 本轮落地优化
- 同步：日志从明细输出改为摘要输出
- 导出：动态脚本加载支持复用、超时清理、已加载标记
- 预览：主页面与微信页弹窗均增加一次性关闭保护
- OCR：增加标题候选清洗与动作型会议主题提取
- 导出链路：index.html 已移除旧 docx 静态脚本，report.js 改为本地优先加载 vendor/docx.8.2.0.umd.cjs，失败后再回退 CDN

## 当前新评估点
- 上传入口是否保留“主页面 + 微信轻量页”双入口，还是在保留兼容性的同时收敛为共用实现

## 当前结构结论
- 保留双入口：index.html 负责完整工作台上传；wechat-upload.html 负责微信环境轻量上传
- 共用逻辑：识别预览、结果摘要、确认交互收敛到 js/upload-flow.js

## 当前新增目标
- 微信页成功保存后自动返回主页面
- 预览中展示每条记录为何新增、合并或跳过

## 本轮补充结果
- 微信上传成功后的返回行为仍在轻量页内完成，不影响主页面上传流程
- 识别依据由 OCR 动作计划生成，经 js/upload-flow.js 渲染到主页面与微信页

## 当前新增目标
- 在预览中展示“匹配到的已有会议摘要”
- 让 OCR 识别后的去重与参会人合并依据更具体、更可核验

## 本轮补充结果
- OCR 动作计划现在同时输出 previewReason、reason、matchedExistingSummary
- js/upload-flow.js 负责把“依据”和“匹配到的已有会议摘要”渲染到主页面与微信页预览中

## 当前新增目标
- 会议面板默认排序要体现局领导优先顺序
- OCR 二次校正层继续增强，提升新增会议排序所依赖的人名识别准确率

## 本轮补充结果
- 会议面板默认排序在未拖动且未完成场景下按领导优先级和时间共同决定
- OCR 二次校正层会在会议入库前清洗参会人员、地点、会议室编号，并统一领导优先排序
- 会议拖拽排序通过 manualOrder 显式区分，只影响真正拖动过的会议

## 当前排查重点
- 继续核对主页面手动新增会议是否也走同一套“领导优先 + 同级按时间”排序链路
- 排查线上仍出现“钱局会议落底”的原因，优先确认排序修复是否完整覆盖 addItem、loadItems、renderColumn 与缓存版本刷新

## 本轮补充结果
- 主页面手动新增会议在保存前会先做参会人员顺序整理，并重置未拖动会议的 order/manualOrder 状态
- js/db.js 现在统一兜底会议存储状态，避免手动新增、OCR 新增、编辑更新时把旧 order 残留回数据库
- js/ocr.js 的二次校正层改为标题、地点、参会人员共享一套领导别名纠偏规则

## 当前新增排查点
- 会议拖动后会把一整列会议写入 manualOrder，因此需要继续核对“后续新增会议”如何插入到这批旧排序之间
- 当前重点不只是清理历史残留 order，还要避免 manualOrder 对后续新增会议形成整列压制

## 本轮新增结果
- 会议排序现在区分“拖动布局生成时间”和“会议创建时间”，新增会议不再被历史 manualOrder 整列压到下方
- 现有拖动结果继续在旧会议之间生效，但拖动后的新会议会重新走“领导优先 + 同级按时间”插入规则

## 当前最新修正
- 会议列排序不再只依赖 sort 比较器，而是改为"按置顶/正常/沉底/已完成分桶，再把新增会议按默认规则合并进手动排序列表"
- 这样可以保证历史手动顺序继续存在，同时新增的钱局、吴局会议能够稳定插入到对应领导层级，而不是被整列压到底部

## 安全优化（第1批已完成）

- 密码存储：btoa() 明文编码已替换为 cryptoManager AES-GCM 256位加密，带 v2 版本标记向后兼容
- API Key：Supabase anon key 添加注释说明安全性依赖 RLS 策略
