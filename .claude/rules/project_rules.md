# 项目协作规则

## 本文件职责
- 只记录本项目长期有效、可跨轮次复用的规则、执行边界和安全网
- 不记录某一轮的临时任务、一次性排查过程和具体改动明细
- 通用协作原则见 universal_template.md，本文件只补充本项目特有要求

## 项目特有长期规则

### 版本号格式（自 v5.1.67 起生效，三段式）
- 格式：`v{月份}.{大版本}.{改动号}`，例如 `v5.1.67` 表示 5 月第 1 个大版本、第 67 次改动
- 代码中写法：`2026-05-11 v5.1.67`（日期 + 版本号）
- 月份变化时改动号重新从 1 开始，例如 6 月第一次为 `v6.0.1`
- **大版本**（序号递增）：app拆分、新增独立功能模块、架构重构、同步引擎重写等
- **小版本**（改动号递增）：Bug修复、UI调整、小功能、性能优化等
- 历史版本 v5.xx 及更早为旧格式，不再使用
- AI 辅助识别：每次改动后自动判断是大版本还是小版本，并更新对应序号

### 业务逻辑相关
- 对 OCR、会议识别、去重、合并类功能，必须同时检查“新增”“合并”“跳过”三种结果是否合理
- 发现回归风险时，优先补充预览、确认、撤销、快照恢复等兜底流程，避免错误数据直接写入面板或云端
- 新增功能或修复问题时，优先同时考虑桌面浏览器、移动端、微信内环境三类场景的一致性
- 涉及办文、会议、待办的“周期性 / 跨日期 / 批量影响”逻辑时，必须同时检查“仅当前”“当前及之后”“全部范围”三类作用范围是否正确
- 涉及同步、导入、识别确认等高风险链路时，优先保证“失败可回退、取消不写入、异常不扩散”

### 代码与结构约束
- 任何代码修改前必须先确认当前文件最新状态，避免覆盖已完成优化
- 严禁把已拆分、已收敛、已安全化的代码还原回旧结构，如把共享模块重新内联、把 DOM API 改回高风险 innerHTML、把安全存储退回明文存储
- 每次修改后都要检查本次改动是否误伤既有优化点，尤其是上传预览、OCR 去重、同步回滚、微信兼容、会议排序、办文跨日期逻辑
- 优先复用现有工具函数、共享模块和既有模式，避免重复造轮子

### 交付与验证要求
- 代码改动后先运行可用的语法检查、诊断或其他校验，再做本地预览和关键路径测试
- 本地模拟真人测试通过后，默认执行提交、推送与部署，不需要额外等待用户再次确认；除非用户明确说本轮不要部署或不要推送
- 每次交付都要明确写出“本次修改后如何测试”，让用户能快速验证改动点
- 每次优化或修复完成后，都要给用户一个运行总结，说明做了什么、影响边界是什么、当前还剩什么风险
- 涉及部署时，必须明确说明推送分支、是否已推送、是否需要等待平台生效、线上应如何强制刷新验证

### 优化方案管理
- 生成的优化方案、阶段性事项和跳过项，必须写入 todolist.md 并实时更新状态
- 用户明确跳过的优化项必须如实记录为“已跳过”，不得自行恢复或猜测需求
- 事故级问题或反复回归问题，在修复后要沉淀为长期规则或迭代记录，避免重复犯错

## 每轮执行边界

### 开始前必做
1. 通读 `.claude/rules/` 下全部规则文件：project_rules.md、project_framework.md、todolist.md、project_iteration_log.md、universal_template.md
2. 若用户指定了额外的标准文件、方案文件或桌面规则文档，先读取再动手
3. 更新 todolist.md，明确本轮目标、风险点、当前待办和验证计划
4. 在 project_iteration_log.md 中新增或补全本轮记录
5. 修改代码前先确认相关文件的当前状态，避免基于旧上下文动手

### 进行中必做
1. 小步修改，避免大面积无关改动
2. 出现异常先查当前实现、历史记录和既有规则，再决定修改策略
3. 若结构、链路或文件职责发生稳定变化，同步更新 project_framework.md
4. 若发现规则缺口、交付反复遗漏或流程性失误，及时补到 rules 中，避免下轮继续忘

### 结束前必做
1. 回写 todolist.md，更新完成情况、验证结果和下一步
2. 回写 project_iteration_log.md，记录本次目标、改动、验证、提交记录和遗留事项
3. 若结构变化已稳定生效，更新 project_framework.md
4. 若涉及代码改动，执行可用的 build / lint / 语法检查 / 诊断 / 关键路径测试
5. 若用户要求或项目场景涉及部署，完成提交、推送与部署，并告知用户线上测试方式

## 安全网规则
- 任何涉及数据清空、覆盖、批量修改、同步写入的操作，都要优先检查是否有备份、撤销或恢复路径
- 任何涉及账号、密码、API Key、云端配置的内容，不得明文写入代码、日志或无保护存储
- 任何涉及跨日期、周期性、批量作用范围的操作，必须验证边界日期和历史数据兼容性，不能只测新建数据

## 数据安全铁律（P3-31 事故教训）

### 绝对禁止
- **禁止在任何改动上线前不做 IndexedDB 数据备份**：每次改动涉及 `db.js`、`sync.js`、`app.js` 中数据读写逻辑时，必须先在浏览器控制台手动导出 IndexedDB 全量数据作为备份，再动手改代码
- **禁止修改渲染层代码时连带改数据层**：UI 渲染改动（如 calendar.js 的卡片结构、样式）绝对不能触碰 `dataset.date`、`dayStates`、`endDate` 等数据字段的写入逻辑，除非有明确的业务需求且已做数据备份
- **禁止在不确定影响范围的情况下推送上线**：改动涉及日历、视图切换、事项渲染时，必须先在本地用**真实数据**完整测试日视图、周视图、月视图三个视图下所有事项类型（待办、会议、办文、跨日期办文、周期性办文）的渲染，确认无丢失后再推送

### 必须执行
- **上线前全量数据校验**：推送前必须在本地浏览器检查 IndexedDB 事项总数与改动前一致，如果数量减少则禁止推送
- **新增功能必须与既有数据完全隔离**：新增的 UI 元素、事件监听、DOM 操作必须通过独立的 class / data 属性 / 事件拦截实现，不得修改既有卡片的任何属性（包括 style.position、draggable、dataset 等），除非该属性仅对新增功能生效
- **改动后立即线上验证数据完整性**：推送后在线上页面用 Ctrl+Shift+R 强刷，检查 IndexedDB 事项总数、各类型数量与改动前一致，发现异常立即 revert

### 事故复盘
- 2026-04-24 P3-31 事故：为周/月视图增加拖拽延伸手柄时，修改了所有 calendar-item 的 `style.position`、click 事件拦截逻辑，并修改了月视图 `dataset.date` 的赋值方式，导致周期性办文在视图中不可见，用户数据丢失且无法恢复
- 根因：渲染层改动越界影响了数据层，且上线前未用真实数据做全量回归验证
- 教训：**渲染层改动绝不碰数据字段，上线前必须校验 IndexedDB 数据完整性**

- 2026-04-29 P3-46 事故：sync.js 使用 clearAllItems+addItem 模式导致同步后所有事项 ID 变化，编辑/删除报"事项不存在"；getItemKey 去重键含 item.id 导致周期性办文同步时重复创建；用户删除重复项后同步把空数据覆盖云端，全部数据丢失；导入备份后同步立即覆盖清空
- 根因：同步链路使用"清空+重建"模式、去重键不稳定、无数据丢失保护、导入未暂停同步
- 教训：**同步链路绝对不能用清空重建模式，必须保留已有 ID；导入必须暂停同步；云端备份必须与同步数据隔离**

## 同步防数据丢失机制（P3-46 起实施，必须长期遵守）
- **putItem 替代 clearAllItems+addItem**：同步时保留已有 ID，避免编辑失败
- **deleteItemsByHashes 按需清理**：仅在云端数据量 ≥ 本地时才清理多余项
- **数据量比保护**：本地≥5条且云端不足30%时阻止下载覆盖
- **导入暂停同步**：importFromFile 期间设 isSyncing=true，导入后自动 uploadToCloud
- **自动备份 20 份**：同步前 autoBackupBeforeSync 保留最近 20 份到 localStorage
- **云端每日备份**：每晚 8 点保存到云端 dailyBackups 字段，滚动 30 份，与同步数据隔离
- **本地每日备份**：每晚 8 点自动下载 JSON 文件到本地
- **uploadToCloud 保留 dailyBackups**：上传时先读取云端已有 dailyBackups 合并回去

## 推送安全铁律（2026-05-04 事故教训）

### 绝对禁止
- **禁止通过 GitHub API（create_or_update_file / push_files）推送任何超过 1KB 的文件**：API 参数 content 字段容易截断，导致线上白屏。index.html（~69KB）被截断到 2.4KB 就是此原因
- **禁止在 git push 失败时用 API 绕过**：如果终端环境的 git push 不生效，必须请用户在独立终端手动执行，绝不能用 API 替代

### 必须执行
- **推送后必须验证远程文件完整性**：通过 GitHub API 检查远程文件的 size 字段，确认与本地一致
- **git push --force 在当前 PowerShell 环境下可能静默失败**：exit code 0 但实际未推送。推送后必须通过 GitHub API 的 list_commits 确认远程 HEAD 已更新

## .claude/rules/ 目录维护（规范目录）
- 固定维护五个文件：project_rules.md、project_framework.md、todolist.md、project_iteration_log.md、universal_template.md
- project_rules.md 只记录长期规则
- project_framework.md 只记录稳定结构与核心链路
- todolist.md 只记录当前轮次的操作面板信息
- project_iteration_log.md 只记录每一轮的正式留痕
- universal_template.md 只记录跨项目可复用的通用模板与原则
- **`.claude/rules/` 为规范目录**，修改后自动同步到 `.trae/rules/`
- 如通过 Trae 修改了 `.trae/rules/`，下次用 Claude Code 时需反向同步回 `.claude/rules/`

## 规则优先级
- 全局规则（`~/.claude/rules/ecc/`）和记忆（`~/.claude/projects/.../memory/`）为最高级别
- 项目规则文件（`.claude/rules/`）仅为辅助——帮助理解项目、跟踪开发内容
- 当项目规则与全局规则/记忆冲突时，以全局规则和记忆为准
