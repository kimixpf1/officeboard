# 通用项目模板

> 适用于所有新项目的基础规则和文件结构模板。每个新项目开始时复制此目录并按实际情况填充。

## .trae/rules/ 目录结构

```
.trae/rules/
├── project_rules.md          # 项目长期规则
├── project_framework.md      # 项目框架描述
├── todolist.md               # 待办事项
├── project_iteration_log.md  # 迭代记录
└── universal_template.md     # 本文件（通用模板）
```

## project_rules.md 通用规则模板

```markdown
# 项目长期规则

- 每次开始工作前先阅读 .trae/rules/ 下的所有规则文件。
- 始终保持大局观，优先保证整条业务链路稳定，不只修单点问题。
- 任何修改都必须以"不破坏现有全部功能"为前提，先兼容、后优化。
- 每次完成改动后，必须做真人模拟测试，再结束本轮工作。
- 完成修改并确认功能无误后，默认执行推送部署。
- 每次动手前先更新 todolist.md 与 project_iteration_log.md；完成后再补齐结果。
- 每次改动完成后，更新项目框架、todolist、迭代记录，写清改动、结构变化、遗留事项和下一步。
- 代码保持简洁高效，不要冗余堆叠。优先复用已有工具函数和组件，避免重复造轮子。
- 项目依赖和工具链不得随意更换，新增依赖需确认兼容且无冗余替代品。
- 目标平台和运行环境约束：（按项目填写，如"手机端微信内打开"或"桌面浏览器"）。
- 部署目标和构建配置：（按项目填写，如"GitHub Pages"或"Vercel"）。
```

## 跨项目通用原则

### 工作流
1. 开工前必读 .trae/rules/ 全部文件，结束后更新 todolist/framework/iteration_log
2. 多步骤任务用 TodoWrite 展示待办并实时更新
3. 推送部署后必告知 commit hash 前7位 + 部署 URL
4. 每次改动后整理线上验证清单供用户验收
5. 验证通过后及时 commit + push，不积压
6. 每次用 GLM 模型后运行 `python "c:\Users\42151\.glm-tools\glm_usage.py" --all` 查两个 Key 用量并报告

### 编码原则
7. 必须通过 Task/Skill 实际调用已安装的智能体和技能
8. 新功能→编排→编码→自查→验证；Bug→定位根因→最小修复
9. 代码改动后自动做安全/类型/错误处理自查
10. 渐进迭代小步快跑，不破坏现有功能，先兼容后优化
11. 复用已有工具函数和模式，不重复造轮子
12. 涉及移动端时兼容窄屏、触控、安全区

### 安全
13. 敏感信息（密码/API Key）不得写入代码或日志，必须加密存储
14. 涉及用户输入/API/认证时做安全扫描

### 规则维护
15. 项目规则只记项目特有内容，通用原则写在此文件，不重不漏
