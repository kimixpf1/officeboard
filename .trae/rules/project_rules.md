# 项目协作规则

## 默认执行原则
- 每次功能修改结束后，先做完整检查，再提交、推送、部署
- 修改时保持大局观，优先保证现有所有功能完好，不因修一个点破坏其他功能
- 交付前尽量做接近真人使用路径的模拟测试，至少覆盖核心主流程、关键弹窗、上传识别、保存确认等场景
- 新增功能或修复问题时，优先考虑移动端、微信内环境、桌面浏览器三类使用场景的一致性
- 对 OCR、会议识别、去重、合并类功能，必须同时检查“新增”“合并”“跳过”三种结果是否合理
- 发现回归风险时，优先补充预览、确认、兜底流程，避免直接写入错误数据
- .trae/rules/ 目录固定维护四个文档：project_framework.md、todolist.md、project_iteration_log.md、project_rules.md；每次动手前后都要同步更新，避免重复大面积读代码和遗漏历史上下文
- 一般使用 Git 维护迭代记录，改完及时提交代码，并在需要时补充到项目迭代记录文档

## 交付流程
- 开始动手前先更新 .trae/rules/ 下的项目框架、todolist、项目迭代记录
- 代码修改后先运行可用的语法检查、诊断或其他校验命令
- 校验通过后再进行本地预览和关键流程测试
- 修改完成后再次更新 .trae/rules/ 下的项目框架、todolist、项目迭代记录
- 确认功能无误后，执行提交、推送与部署

## 沟通原则
- 默认用中文回复
- 给出结论时说明已验证的范围与未验证的边界
- 涉及部署时，明确说明推送分支、是否已部署、是否需要等待平台生效
- 每次交付都要明确写出“本次修改后用户该如何测试”，让用户能快速验证改动点是否正常
